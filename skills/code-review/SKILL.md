# コードレビュースキル

## レビュー優先度

### 🔴 CRITICAL（マージブロック）
- **セキュリティ**: 脆弱性、秘密情報の漏洩、認証・認可の問題
- **正確性**: ロジックエラー、データ破壊リスク、競合状態
- **破壊的変更**: バージョニングなしのAPIコントラクト変更
- **データ損失**: データ喪失・破損のリスク

### 🟡 IMPORTANT（要議論）
- **コード品質**: SOLID原則の重大な違反、過度な重複
- **テストカバレッジ**: クリティカルパスや新機能のテスト不足
- **パフォーマンス**: 明らかなボトルネック（N+1クエリ、メモリリーク等）
- **アーキテクチャ**: 確立されたパターンからの大幅な逸脱

### 🟢 SUGGESTION（ノンブロッキング改善提案）
- **可読性**: 不適切な命名、簡略化可能な複雑ロジック
- **最適化**: 機能影響のないパフォーマンス改善
- **ベストプラクティス**: 規約からの軽微な逸脱
- **ドキュメント**: 不足・不完全なコメント/ドキュメント

## レビュー原則

1. **具体的に**: 正確な行番号、ファイル、具体例を参照する
2. **文脈を提供**: なぜ問題なのか、潜在的な影響を説明する
3. **解決策を提示**: 問題指摘だけでなく修正コードも示す
4. **建設的に**: コード改善に焦点を当て、著者を批判しない
5. **良い点も認める**: 良いコードやスマートな解決策を称賛する
6. **実用的に**: すべての提案を即座に適用する必要はない

## チェック項目

### クリーンコード
- 変数、関数、クラスに意味のある命名がされているか
- 単一責任の原則: 各関数/クラスが一つのことをうまくこなしているか
- DRY: コードの重複がないか
- 関数が小さく焦点が絞られているか（理想は20-30行以内）
- 深いネストがないか（最大3-4レベル）
- マジックナンバー・マジックストリングがないか（定数を使用）

### エラーハンドリング
- 適切なレベルでエラーハンドリングされているか
- 意味のあるエラーメッセージが設定されているか
- サイレント失敗や無視された例外がないか
- Fail Fast: 入力バリデーションが早期に行われているか

### セキュリティ
- パスワード、APIキー、トークン、個人情報がコードやログに含まれていないか
- ユーザー入力がバリデーション・サニタイズされているか
- パラメータ化クエリが使用されているか（SQLインジェクション対策）
- 認証・認可チェックが適切か

### テスト品質
- クリティカルパスと新機能にテストがあるか
- テスト名が何をテストしているか説明的か
- Arrange-Act-Assert パターンが明確か
- エッジケース（境界値、null、空コレクション）がテストされているか

### パフォーマンス
- N+1クエリがないか
- 適切な時間/空間計算量か
- キャッシュが活用されているか
- リソースの適切なクリーンアップ
- 大量データのページネーション対応

### React パフォーマンス（React コード対象時）
- **CRITICAL**: ウォーターフォール（逐次await）がないか → `Promise.all()` で並列化
- **CRITICAL**: バレルファイル経由のimportがないか → 直接importに修正
- **HIGH**: 不必要な再レンダリングの原因がないか（インラインコンポーネント定義、effectでの派生state計算など）
- **MEDIUM**: 動的importが活用されているか（重いコンポーネントの遅延ロード）
- **MEDIUM**: 条件レンダリングが `&&` ではなく三項演算子を使っているか
- 詳細ルールは `skills/react-performance/` を参照

### UI/UXデザイン品質（UI関連コード対象時）
- アクセシビリティ: セマンティックHTML、キーボード操作、ARIA属性、カラーコントラスト
- レスポンシブ: モバイルファースト、適切なブレークポイント、タッチターゲットサイズ
- フォーム: ラベル表示、インラインバリデーション、エラーメッセージ
- フィードバック: ローディング状態、成功/失敗通知、楽観的更新
- 詳細ルールは `skills/ui-design/` を参照
