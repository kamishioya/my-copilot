# 体系的デバッグ

## 概要

ランダムな修正は時間を浪費し、新たなバグを生む。応急処置は根本的な問題を隠す。

**核心原則:** 修正を試みる前に、必ず根本原因を特定する。症状の修正は失敗である。

## 鉄則（Iron Law）

```
根本原因の調査なしに修正を行わない
```

フェーズ1を完了していなければ、修正を提案できない。

## 使用場面

あらゆる技術的問題に使用:
- テスト失敗
- 本番バグ
- 予期しない動作
- パフォーマンス問題
- ビルド失敗
- 統合問題

**特に以下の場合に使用:**
- 時間的プレッシャーがあるとき（緊急時ほど推測の誘惑が大きい）
- 「ちょっとした修正」が明白に見えるとき
- すでに複数の修正を試みたとき
- 前回の修正がうまくいかなかったとき

## 4つのフェーズ

各フェーズを順に完了すること。

### フェーズ1: 根本原因の調査

**修正を試みる前に:**

#### 1. エラーメッセージを注意深く読む
- エラーや警告を読み飛ばさない
- 多くの場合、正確な解決策が含まれている
- スタックトレースを最後まで読む
- 行番号、ファイルパス、エラーコードをメモする

#### 2. 安定して再現する
- 確実にトリガーできるか？
- 正確な再現手順は？
- 毎回発生するか？
- 再現できない場合 → データを集める（推測しない）

#### 3. 最近の変更を確認する
- 原因となりうる変更は？
- git diff、最近のコミット
- 新しい依存関係、設定変更
- 環境の違い

#### 4. マルチコンポーネントシステムでの証拠収集

システムが複数のコンポーネントを持つ場合:

```
各コンポーネント境界について:
  - コンポーネントに入るデータをログ
  - コンポーネントから出るデータをログ
  - 環境/設定の伝播を検証
  - 各レイヤーの状態を確認

1回実行して、どこで壊れるか示す証拠を収集
→ 証拠を分析して故障コンポーネントを特定
→ そのコンポーネントを重点的に調査
```

**.NET の例:**
```csharp
// レイヤー1: API Controller
_logger.LogDebug("=== リクエスト受信: {Method} {Path} ===", 
    Request.Method, Request.Path);
_logger.LogDebug("=== ヘッダー: {Headers} ===", 
    string.Join(", ", Request.Headers.Select(h => $"{h.Key}={h.Value}")));

// レイヤー2: Service
_logger.LogDebug("=== Service入力: {Input} ===", 
    JsonSerializer.Serialize(input));

// レイヤー3: Repository
_logger.LogDebug("=== SQL実行: {Query}, パラメータ: {Params} ===", 
    query, JsonSerializer.Serialize(parameters));
```

#### 5. データフローを追跡する

- 不正な値はどこで発生するか？
- 誰がこの不正な値でこれを呼び出したか？
- ソースが見つかるまで遡り続ける
- ソースで修正する（症状の場所ではない）

### フェーズ2: パターン分析

**修正前にパターンを見つける:**

1. **動作する例を見つける** — 同じコードベース内の類似する動作コード
2. **リファレンスと比較する** — リファレンス実装を完全に読む（スキミングしない）
3. **差異を特定する** — 動作するものと壊れたものの違いを列挙
4. **依存関係を理解する** — 他に何が必要か、どの設定・環境が前提か

### フェーズ3: 仮説とテスト

**科学的方法:**

1. **単一の仮説を立てる**
   - 明確に述べる: 「Yという理由でXが根本原因だと考える」
   - 書き留める
   - 具体的に（曖昧にしない）

2. **最小限のテスト**
   - 仮説をテストする最小の変更を行う
   - 一度に一つの変数のみ
   - 複数のものを同時に修正しない

3. **続行前に検証**
   - うまくいった → フェーズ4へ
   - うまくいかなかった → 新しい仮説を立てる
   - 上に追加の修正を重ねない

### フェーズ4: 実装

**症状ではなく根本原因を修正する:**

#### 1. 失敗するテストケースを作成する
- 最もシンプルな再現
- 可能なら自動テスト
- 修正前に必ず作成する

#### 2. 単一の修正を実装する
- 特定された根本原因に対処
- 一度に一つの変更
- 「ついでに」の改善はしない
- リファクタリングを混ぜない

#### 3. 修正を検証する
- テストが通るか？
- 他のテストは壊れていないか？
- 問題は実際に解決されたか？

#### 4. 修正がうまくいかない場合
- **停止する**
- 何回修正を試みたか数える
- 3回未満: フェーズ1に戻り、新情報を元に再分析
- **3回以上: 停止してアーキテクチャを疑問視する（下記参照）**

#### 5. 3回以上の修正が失敗した場合：アーキテクチャを疑う

**アーキテクチャ上の問題を示すパターン:**
- 各修正が異なる場所の新しい共有状態/結合/問題を暴く
- 修正に「大規模なリファクタリング」が必要
- 各修正が別の場所に新しい症状を生む

**基本に立ち返って疑問視する:**
- このパターンは根本的に健全か？
- 慣性でこのアプローチに固執していないか？
- 症状を修正し続けるよりアーキテクチャを変更すべきか？

**これ以上の修正を試みる前にユーザーと議論する。**

## 多層バリデーション（Defense-in-Depth）

バグを修正した後、同じバグを構造的に不可能にする:

### レイヤー1: エントリーポイントバリデーション
```csharp
public async Task<Project> CreateAsync(string name, string workingDirectory)
{
    if (string.IsNullOrWhiteSpace(workingDirectory))
        throw new ArgumentException("workingDirectory は空にできません");
    if (!Directory.Exists(workingDirectory))
        throw new DirectoryNotFoundException($"ディレクトリが存在しません: {workingDirectory}");
    // ...続行
}
```

### レイヤー2: ビジネスロジックバリデーション
```csharp
public async Task InitializeWorkspace(string projectDir, string sessionId)
{
    if (string.IsNullOrEmpty(projectDir))
        throw new InvalidOperationException("ワークスペース初期化には projectDir が必要です");
    // ...続行
}
```

### レイヤー3: 環境ガード
```csharp
public async Task GitInit(string directory)
{
    if (Environment.GetEnvironmentVariable("ASPNETCORE_ENVIRONMENT") == "Test")
    {
        var normalized = Path.GetFullPath(directory);
        var tmpDir = Path.GetFullPath(Path.GetTempPath());
        if (!normalized.StartsWith(tmpDir))
            throw new InvalidOperationException(
                $"テスト中にtempディレクトリ外でgit initを拒否: {directory}");
    }
    // ...続行
}
```

### レイヤー4: デバッグ計装
```csharp
_logger.LogDebug("git init実行直前: directory={Directory}, cwd={Cwd}", 
    directory, Environment.CurrentDirectory);
```

**すべてのレイヤーが必要。** 一つのバリデーションポイントで止めない。

## 根本原因追跡テクニック

### コールスタックを遡る

1. **症状を観察する** — エラーが発生する箇所を特定
2. **直接的な原因を見つける** — そのコードを直接引き起こすのは何か
3. **何がこれを呼び出したか追う** — コールスタックを1レベルずつ遡る
4. **遡り続ける** — 不正な値の起源を見つけるまで
5. **ソースで修正する** — 症状の場所ではなく

**重要原則: エラーが表示される場所だけで修正しない。** 元のトリガーまで遡って修正する。

## 危険信号 — 停止してプロセスに従う

以下を考えている自分に気づいたら:
- 「今はとりあえず直して、後で調査する」
- 「Xを変えて動くか見てみよう」
- 「複数の変更を入れてテストを回す」
- 「テストはスキップして手動で確認する」
- 「多分Xだ、それを直そう」
- 「完全には理解していないがこれで動くかも」
- データフローを追跡する前に解決策を提案する
- 「もう1回修正を試す」（すでに2回以上試した場合）

**上記すべて → 停止。フェーズ1に戻る。**

## よくある合理化と現実

| 言い訳 | 現実 |
|--------|------|
| 「問題はシンプル、プロセス不要」 | シンプルな問題にも根本原因がある。プロセスはシンプルなバグには高速 |
| 「緊急、プロセスの時間がない」 | 体系的デバッグは推測と試行錯誤より速い |
| 「まずこれを試して、それから調査」 | 最初の修正がパターンを決める。最初から正しくやる |
| 「修正が動くか確認してからテストを書く」 | テストなしの修正は定着しない。テストが先に証明する |
| 「複数修正を同時にすると時短」 | 何が効いたか分離できない。新しいバグを生む |

## クイックリファレンス

| フェーズ | 主な活動 | 成功基準 |
|---------|---------|---------|
| **1. 根本原因** | エラー読解、再現、変更確認、証拠収集 | 何がなぜ起きたか理解 |
| **2. パターン** | 動作する例を発見、比較 | 差異を特定 |
| **3. 仮説** | 理論を立て、最小限でテスト | 確認済みまたは新仮説 |
| **4. 実装** | テスト作成、修正、検証 | バグ解決、テスト通過 |

## 実績データ

- 体系的アプローチ: 15〜30分で修正
- ランダム修正アプローチ: 2〜3時間の試行錯誤
- 初回修正成功率: 95% vs 40%
- 導入される新バグ: ほぼゼロ vs 頻発
