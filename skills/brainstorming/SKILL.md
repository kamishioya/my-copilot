# ブレインストーミング — アイデアから設計へ

## 概要

アイデアを協調的な対話を通じて、完全に形成された設計と仕様に変換する。

プロジェクトのコンテキストを理解し、一度に一つずつ質問して、構築すべきものへの理解を深める。
設計を理解したら、提案してユーザーの承認を得る。

## ハードゲート

```
設計を提示し、ユーザーが承認するまで、
いかなる実装スキルの呼び出し、コードの記述、プロジェクトのスキャフォールドも行わない。
これはプロジェクトの見かけのシンプルさに関係なく、すべてに適用される。
```

## アンチパターン:「これはシンプルすぎて設計不要」

すべてのプロジェクトがこのプロセスを通る。TODOリスト、単一関数のユーティリティ、設定変更 — すべて。
「シンプル」なプロジェクトこそ、検証されていない前提が最も多くの無駄な作業を生む場所。
設計は短くてよいが（本当にシンプルなプロジェクトなら数文）、必ず提示して承認を得る。

## プロセスフロー

### ステップ1: プロジェクトコンテキストの探索
- 関連ファイル、ドキュメント、最近のコミットを確認
- 既存の技術スタック（.NET、React、MAUI等）を把握
- 既存のパターンや規約を理解

### ステップ2: 明確化のための質問
- **一度に一つの質問**に集中する
- 目的、制約、成功基準を理解する
- 技術的な前提条件を確認する
- ユーザーの優先事項を把握する

質問の例:
- 「この機能の主な利用者は誰ですか？」
- 「既存の○○との関係はどうなりますか？」
- 「パフォーマンスの要件はありますか？」
- 「エラー時にユーザーにどう伝えるべきですか？」

### ステップ3: 2〜3のアプローチを提案
- 各アプローチのトレードオフを明示
- 推奨案を示す（理由とともに）
- .NET/React/MAUI の技術特性を考慮

提案フォーマット:
```markdown
## アプローチ比較

### アプローチA: [名前]
- **メリット**: ...
- **デメリット**: ...
- **適切な場面**: ...

### アプローチB: [名前]
- **メリット**: ...
- **デメリット**: ...
- **適切な場面**: ...

### 推奨: アプローチA
**理由**: ...
```

### ステップ4: 設計の提示
- 複雑さに応じてセクションごとに提示
- 各セクションでユーザーの承認を得る
- シンプルな場合は数文で十分

設計に含めるべき要素:
- **目的と範囲**
- **主要コンポーネント**
- **データフロー**
- **UI/UX の概要**（該当する場合）
- **技術的な決定事項**
- **エッジケースと制約**
- **セキュリティ考慮事項**

### ステップ5: 設計書の作成
- `docs/spec/` に保存
- 命名規則: `YYYY-MM-DD-<トピック>-design.md`
- 仕様書作成スキル（create-spec）のテンプレートに従う

### ステップ6: 実装計画への移行
- 設計が承認されたら、実装計画作成スキル（create-plan）を使用
- 設計書を参照しながら具体的なタスクに分解

## 質問のガイドライン

### 良い質問
- 具体的で文脈に基づいている
- ユーザーの意図を深く掘り下げる
- 技術的な選択肢を提示しながら確認する

### 避けるべき質問
- 一度に複数の質問をすr
- はい/いいえでしか答えられない閉じた質問
- 仮定に基づいた誘導質問

## .NET + React プロジェクトでの考慮事項

- フロントエンド（React）とバックエンド（.NET）の責任分担
- API契約の設計（REST / GraphQL）
- 認証・認可のフロー
- 状態管理の戦略（Redux、React Query等）
- サーバーサイドレンダリング vs クライアントサイドレンダリング

## .NET MAUI プロジェクトでの考慮事項

- 対象プラットフォーム（Android、iOS、Windows、macOS）
- MVVM パターンの適用
- プラットフォーム固有機能 vs 共通コード
- オフライン対応の要件
- デバイスリソース（カメラ、GPS等）の利用
- アプリサイズとパフォーマンスの考慮

## 危険信号 — 停止する

- 設計承認前にコードを書き始めている
- 質問せずに前提で進めている  
- 「シンプルだから設計不要」と考えている
- ユーザーの意図を確認せずに技術選定している
- 一つのアプローチしか検討していない01

**上記すべて → 停止。ステップ2に戻る。**

## 最重要

設計は実装の10倍安い。
設計の間違いを修正するのは、コードの間違いを修正するより桁違いに安い。

**急がば回れ — まず設計、それから実装。**
