# 再レンダリングとstate設計

## 優先度

HIGH

## 見る場所

- 入力フォーム
- 検索、絞り込み、ソート
- 大きい一覧やカード群
- 親stateの変化で広く再描画される画面

## ルール

### 1. コンポーネントをコンポーネント内で定義しない

- 毎回別コンポーネントとして扱われ、再マウントを招く
- 必要な値はpropsで渡す

### 2. 派生stateは基本的にレンダリング中に計算する

- 件数、フラグ、選択状態などは `effect` + `setState` で複製しない
- 元stateから同期計算できるならその場で計算する

### 3. 高コストな部分だけ分離する

- 重い一覧やグラフだけを子へ分離し、必要なら `memo` を使う
- 全体を雑にメモ化するより、境界を小さく切るほうが再現性が高い

### 4. 高頻度更新には適切な道具を使う

- 非緊急更新は `startTransition`
- 重い派生描画は `useDeferredValue`
- 描画に不要な一時値は `useRef`

### 5. 初期化コストが高いstateは遅延初期化する

- `useState(() => createInitialValue())` の形にする

### 6. 前の値だけに依存する更新は関数型 `setState` を使う

- コールバック安定化と依存削減に効く

## 指摘してよい条件

- 親の軽い更新で重い子が毎回動く
- 派生stateを `effect` で複写している
- 入力のたびに大きな一覧や高コスト処理が即同期で走る

## 指摘してはいけない条件

- 計算が軽く、メモ化コストのほうが高い
- 子を分離すると責務や可読性が大きく悪化する
- `memo` を入れても props が毎回新しく、境界改善がない

## 最小修正の考え方

- まず state を減らす
- 次に描画境界を切る
- それでも重い部分だけ `memo` / `startTransition` / `useDeferredValue` を使う
