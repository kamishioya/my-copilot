# 適用範囲と非目標

## ランタイム別の優先対象

### React SPA + .NET API

- 最優先: ウォーターフォール、初期バンドル、再レンダリング、effectの整理
- 条件付き: プリロード、遅延ロード、イベント最適化
- 原則不要: RSC、ハイドレーション、Server Action 前提の提案

### Next.js / SSR / RSC

- SPA向けの項目に加えて、サーバー側の並列fetch、シリアライズ量、ハイドレーション差分も見る
- クライアントへ渡すデータは最小化する
- サーバーcacheは重複取得や高価な計算に限定して使う

## 非目標

- `useMemo` `useCallback` をとりあえず追加すること
- 小さなコンポーネントまで機械的に `memo` 化すること
- すべてを dynamic import に分割すること
- 依存追加で複雑な最適化DSLを持ち込むこと
- 計測なしにキャッシュ層を増やすこと

## AIが明示すべき前提

- この提案は SPA 前提か、SSR 前提か
- 体感改善を狙うのか、計測済みボトルネックを潰すのか
- 構造改善で十分か、追加APIや依存が必要か

## 推奨する出力の形

1. 影響が大きい順に並べる
2. 各項目で「なぜ遅いか」を構造で説明する
3. 変更は最小差分で提案する
4. フレームワーク依存の提案は前提条件を添える
