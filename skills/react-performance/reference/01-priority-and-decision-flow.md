# React パフォーマンス優先順位

## 目的

最初に見る場所を固定し、効果の薄い最適化へ流れないようにする。

## 判定順

1. ネットワーク待ちが直列化していないか
2. 初期表示に不要なコードを積んでいないか
3. 入力やスクロールで高頻度更新が詰まっていないか
4. `effect` が状態計算やイベントロジックを肩代わりしていないか
5. DOM量や描画条件が無駄に重くなっていないか
6. それでも残る場合だけ JavaScript のホットパスを見る

## まず集める事実

- 遅いのは初期表示か、操作中か、一覧描画か
- 問題はSPAか、SSR/Next.jsか
- 同じ関数内に独立した `await` が連続していないか
- 重いライブラリや大型コンポーネントが初回描画に含まれていないか
- 親のstate変更で広い範囲が再描画されていないか

## 指摘に含めるべき情報

- 症状: 何が遅くなるか
- 原因: どの構造がボトルネックか
- 最小修正: 何をどう変えるか
- 適用条件: どのランタイムで有効か
- 除外条件: 依存関係や副作用の順序上、変えてはいけない理由があるか

## 避けること

- 計測も根拠もなく `useMemo` を増やす
- 小さいコンポーネントまで動的 import にする
- 冷たいコードパスに細かいループ最適化を入れる
- Next.js前提の提案をSPAへそのまま出す
