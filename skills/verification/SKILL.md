# 完了前検証

## 概要

検証なしに作業完了を主張することは、効率ではなく不誠実である。

**核心原則:** 常に、主張の前に証拠を。

## 鉄則（Iron Law）

```
最新の検証証拠なしに完了を宣言しない
```

このメッセージ内で検証コマンドを実行していなければ、「通る」とは言えない。

## ゲート関数

```
ステータスの主張や満足の表明の前に:

1. 特定する: この主張を証明するコマンドは何か？
2. 実行する: 完全なコマンドを実行（新規、完全）
3. 読む:    出力全体、終了コード、失敗数を確認
4. 検証する: 出力は主張を裏付けるか？
   - いいえ → 証拠とともに実際の状態を報告
   - はい  → 証拠とともに主張する
5. そのときだけ: 主張する

いずれかのステップをスキップ = 検証ではなく嘘
```

## 一般的な失敗パターン

| 主張 | 必要な証拠 | 不十分なもの |
|------|-----------|-------------|
| テストが通る | テストコマンド出力: 0 failures | 前回の実行結果、「通るはず」 |
| ビルドが成功 | ビルドコマンド: exit 0 | リンター通過で推定 |
| バグが修正された | 元の症状のテスト: 通過 | コード変更のみで修定を推定 |
| 要件を満たした | 行ごとのチェックリスト確認 | テスト通過のみ |

## 技術スタック別の検証コマンド

### .NET
```bash
# テスト実行
dotnet test --verbosity normal

# ビルド確認
dotnet build --no-restore

# 特定プロジェクトのテスト
dotnet test ./tests/MyProject.Tests/ --filter "ClassName=MyTests"
```

### React
```bash
# テスト実行
npx vitest run

# ビルド確認
npm run build

# リンター
npx eslint src/
```

### .NET MAUI
```bash
# ビルド確認（対象プラットフォーム指定）
dotnet build -f net8.0-android
dotnet build -f net8.0-ios

# テスト実行
dotnet test ./tests/MyMauiApp.Tests/
```

## 危険信号 — 停止する

- 「通るはず」「おそらく」「〜のようだ」を使おうとしている
- 検証前に満足を表明（「完璧！」「できた！」など）
- 検証なしにコミット/プッシュ/PRしようとしている
- 部分的な検証に頼っている
- 「今回だけ」と考えている
- 疲れていて作業を終わらせたい
- **検証を実行せずに成功を暗示する表現**

## よくある合理化と現実

| 言い訳 | 現実 |
|--------|------|
| 「もう動くはず」 | 検証を実行する |
| 「自信がある」 | 自信 ≠ 証拠 |
| 「今回だけ」 | 例外なし |
| 「リンターは通った」 | リンター ≠ コンパイラ ≠ テスト |
| 「疲れた」 | 疲労 ≠ 免除 |
| 「部分的なチェックで十分」 | 部分的は何も証明しない |

## キーパターン

### テスト検証
```
✅ [テストコマンド実行] [確認: 34/34 通過] 「全テスト通過」
❌ 「もう通るはず」 / 「正しそう」
```

### リグレッションテスト（TDD Red-Green）
```
✅ 書く → 実行（通過）→ 修正を元に戻す → 実行（失敗するはず）→ 修正復元 → 実行（通過）
❌ 「リグレッションテストを書いた」（Red-Greenサイクルの検証なし）
```

### ビルド検証
```
✅ [ビルド実行] [確認: exit 0] 「ビルド通過」
❌ 「リンターが通ったからビルドも大丈夫」
```

### 要件検証
```
✅ 計画を再読 → チェックリスト作成 → 各項目検証 → ギャップまたは完了を報告
❌ 「テストが通ったからフェーズ完了」
```

## 適用タイミング

**以下の前に常に適用:**
- 成功/完了の主張のあらゆるバリエーション
- 満足の表明
- 作業状態についてのポジティブな声明
- コミット、PR作成、タスク完了
- 次のタスクへの移行

## 最重要

**検証に近道はない。**

コマンドを実行する。出力を読む。**それから**結果を主張する。

これは交渉不可。
